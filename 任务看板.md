第一部分：前期准备与学习资料（重点是快速上手）
•框架基础 (Spring Boot)：
￮推荐去 B 站搜类似“Spring Boot 快速入门 2小时”的视频（比如尚硅谷、黑马的短教程），只要懂了 @RestController, @Service, @Autowired 这几个核心注解，以及怎么连数据库、写基础的增删改查（CRUD）就行。
￮喜欢看文档的话可以直接看官方文档
•协作工具 (Git)：
￮廖雪峰的 Git 教程（看前几章：建库、提交、分支管理、解决冲突）。
￮B 站搜个 10 分钟的“Git 团队协作实战”演示。
•测试工具： 装个 Postman 或者 ApiFox，学会怎么发 GET 和 POST 请求测自己的接口。

第二部分：项目分工与协作设计
你们某个人在 Gitee 或 GitHub 上建个空仓库，搭好一个最基础的 Spring Boot 空架子（带好 pom.xml 和 application.yml），然后分别在这个基础上拉自己的分支进行开发。后面开发的差不多之后在把代码合并。形成完整的功能。
项目拆解：
•本科生 A（负责用户模块）： 实现用户的注册、登录（简单的账号密码比对即可，先不搞 Token 等复杂校验）、查询用户列表。
•本科生 B（负责任务模块）： 实现任务的发布、查看所有任务、修改任务状态（比如从“待办”改成“已完成”）。
第三部分：一周时间线安排
自己安排就行，以后端学习为主，最终会做出一个后端小项目，带一个接口文档（实战中前后端对接用），后端完成的话也可以把接口和项目描述给ai，让ai生成完整的前端项目，让这个项目更完善。


实验室任务看板 (Task Board) - 练手项目需求书
项目目标： 用一周时间，基于 Spring Boot 跑通一个最基础的后端项目，掌握简单的增删改查（CRUD），并能在 Git 团队协作中“活下来”。
总体约束与暗坑：
两人必须在一个统一的 Git 仓库下开发。
接口返回给前端的数据格式，必须统一封装成一个 Result<T> 类（至少包含 code, message, data 三个字段）。注意：你们俩需要商量好这个类怎么写，放到哪个包下，这是你们必踩的 Git 冲突点，提前做好心理准备。

第一部分：数据库表设计 (MySQL)
建两张最简单的表就行，前期为了降低难度，不需要强制设外键约束，靠代码逻辑保证对应即可。
1.用户表 (sys_user)
•id (int, 主键, 自增)
•username (varchar, 账号)
•password (varchar, 密码。前期练手存明文即可，不用搞复杂的加密)
•create_time (datetime, 注册时间)
2.任务表 (lab_task)
•id (int, 主键, 自增)
•title (varchar, 任务标题，例如 "整理实验室桌面")
•content (varchar, 任务详情)
•status (int, 状态：0-待办，1-进行中，2-已完成)
•assignee_id (int, 负责人的用户 ID，关联 sys_user 表的 id)
•create_time (datetime, 创建时间)

第二部分：接口需求与分工
【同学 A 的任务：用户模块】
你要负责大家的账号权限，别让没注册的人进来。
接口 1：用户注册
•请求路径： POST /api/user/register
•需求描述： 接收传来的账号密码，存入数据库。如果查询发现账号已经存在，要返回错误提示“该账号已存在”。
接口 2：用户登录
•请求路径： POST /api/user/login
•需求描述： 接收账号密码去数据库比对。登录成功返回该用户的 id 和 username，失败返回“账号或密码错误”。（因为是新手，先不用做 JWT Token 或 Session 拦截器，业务逻辑走通就行）。
接口 3：获取用户列表
•请求路径： GET /api/user/list
•需求描述： 查出所有注册用户的列表，给后续发任务时选择“负责人”用。
【同学 B 的任务：任务模块】
你要负责看板的核心逻辑，让大家能发任务、改状态。
接口 1：发布任务
•请求路径： POST /api/task/create
•需求描述： 接收传来的标题、内容和负责人ID (assignee_id)，存入数据库，新增的任务默认状态 status 为 0（待办）。
接口 2：查看任务列表
•请求路径： GET /api/task/list
•需求描述： 查询出所有的任务列表返回。
•进阶要求（选做）： 看看能不能在查出任务的同时，把负责人的名字 (username) 也一并查出来返回？（提示：可以通过 assignee_id 去查用户表，或者写个简单的 SQL JOIN 语句）。
接口 3：更新任务状态
•请求路径： PUT /api/task/status/{taskId}
•需求描述： 传一个任务的 id 和新的状态码进来，更新数据库里这条任务的 status 字段（比如把 0 变成 2）。

第三部分：Git 协作演练指南
你们俩千万不要一开始就在同一个分支上写代码！
1.拉取骨架： 你们中的某个人（自己商量）会把包含 pom.xml 和基础依赖的空骨架推送到仓库，你们俩各自 git clone 到本地。（不要开发了一般再建仓库，先建仓库，把骨架放上去再开发）
2.独立分支：
￮A 同学切出一个分支：git checkout -b feature/user
￮B 同学切出一个分支：git checkout -b feature/task
3.制造与解决冲突： 你们在开发时肯定都会修改配置文件（比如连数据库的 application.yml），或者那个公共的 Result.java。当最后要把代码合并到主分支时，一定会报冲突（Merge Conflict）。
4.合并验收： 遇到冲突别慌，两人坐到一起，用 IDEA 的图形化界面（或者直接看代码里的 <<<<<<< HEAD 标记），商量好保留谁的代码。最终主分支跑起来，不报错，才算初步过关。
最终联调标准： 用 Postman 或 ApiFox 建一个集合，把这 6 个接口都配进去。跑在同一个机器上，如果 A 能调通 B 的接口，B 能调通 A 的接口，这周的实战任务就算圆满结束。
